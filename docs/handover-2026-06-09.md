### 2. NUI cache gotcha (important)

The NUI router caches page elements at the JavaScript level (`createRouter()` in `modules/nui_wc2/NUI/nui.js`). `init(element, params, nui)` only runs **once per page per JS context** — subsequent navigations call `element.show?.(params)`. If the first `init()` throws, the cached element stays broken across reloads in the same tab.

**Workaround:** navigate via Projects → click card → Editor → Render & Play to force a fresh init chain. Closing and re-opening the tab also works. The dev-tools "Disable cache" checkbox is the most reliable way to avoid this.

### 1. `<msg>.<split> <label>` header numbering

The user asked: for split messages, show slide headers as `1.0 glm5-chat`, `1.1 glm5-chat`, `1.2 glm5-chat` etc. so it's clear which sub-slides belong to the same source message.

**Design intent:**
- `originalIdx` is the source message index (0-based in the array, displayed as 1-based in the UI: 1, 2, 3, …)
- `splitIdx` is the position within the message (0-based, displayed as 0, 1, 2, …)
- For non-split messages the header just shows `1.0 glm5-chat` (the .0 is always shown so the format is consistent)
- For setup/details/title/end, the header just shows `Setup` / `Details` / `Title` / `End` (no number)
- Applied both in the **player header** and the **slides list row** for consistency

**Why the LLM-removal was right:** the `splitIdx` doesn't need to be stored on the slide. It can be computed at render time by walking the deck once and counting how many previous slides share the same `originalIdx`. This is what I had implemented as a memoized `getSplitIndices()` helper.

**Implementation sketch:**
```javascript
// In web/js/pages/render.js, inside init() or as a top-level function
let splitIndexCache = null;
function getSplitIndices() {
    if (splitIndexCache) return splitIndexCache;
    const map = new Map();
    const seen = new Map(); // originalIdx → running count
    deck.slides.forEach((slide, idx) => {
        if (slide.type !== 'conversation' && slide.type !== 'narration') return;
        if (slide.originalIdx == null) return;
        const count = seen.get(slide.originalIdx) || 0;
        map.set(idx, count);
        seen.set(slide.originalIdx, count + 1);
    });
    splitIndexCache = map;
    return map;
}
```

Then in `loadSlide()`:
```javascript
const splitMap = getSplitIndices();
const splitIdx = splitMap.get(idx) || 0;
let headerLabel;
if (slide.type === 'conversation' && slide.originalIdx != null) {
    const major = slide.originalIdx + 1;
    headerLabel = `${major}.${splitIdx} ${slide.label || slide.speaker}`;
}
```

And in `renderSlideList()` for the row label, swap `${slide.label} #${idx+1}` for `${major}.${splitIdx} ${slide.label}`.

Cache invalidation: clear `splitIndexCache` whenever `deck.slides` is mutated (the `loadProject`, `renderSingleSlide`, and `renderAllSlides` paths).

The `formatTimestamp(iso)` helper returns a two-line-ish "Jun 7, 2026 · 19:37:10" string. Helper definition:
```javascript
function formatTimestamp(iso) {
    if (!iso) return null;
    const d = new Date(iso);
    if (isNaN(d.getTime())) return null;
    const months = ['Jan', 'Feb', 'Mar', 'Apr', 'May', 'Jun', 'Jul', 'Aug', 'Sep', 'Oct', 'Nov', 'Dec'];
    const date = `${months[d.getMonth()]} ${d.getDate()}, ${d.getFullYear()}`;
    const time = `${String(d.getHours()).padStart(2, '0')}:${String(d.getMinutes()).padStart(2, '0')}:${String(d.getSeconds()).padStart(2, '0')}`;
    return `${date} · ${time}`;
}
```

### 3. `*word*` → `word` stripping in TTS / word spans

The user pointed out that audio of `*feel*` is rendered as "asterisk feel asterisk" by nSpeech. The fix: strip `*` markers from the **spoken text** only (the on-screen text keeps them verbatim).

**Implementation sketch:**
```javascript
// Top-level helper in render.js
function stripEmphasisForSpeech(s) {
    if (!s) return s;
    return s.toString().replace(/\*+/g, '');
}

// In getSpokenText():
function getSpokenText(slide) {
    let text = (slide.type === 'title' || slide.type === 'end')
        ? (slide.narration || slide.text || '')
        : (slide.text || slide.narration || '');
    return stripEmphasisForSpeech(text);
}

// In buildWordSpans():
function buildWordSpans(text, tts) {
    const timedSegments = getTimedSegments(tts);
    if (timedSegments.length > 0) {
        return timedSegments.map(segment => {
            const words = segment.words.map(w => {
                const clean = stripEmphasisForSpeech(w.word);
                return `<span class="word future" data-start="${w.startMs}" data-end="${w.endMs}">${escapeHtml(clean)}</span> `;
            }).join('');
            return `<span class="segment future" data-start="${segment.startMs}" data-end="${segment.endMs}">${words}</span> `;
        }).join('');
    }
    if (!text) return '';
    const textWords = stripEmphasisForSpeech(text).split(/\s+/).filter(w => w.length > 0);
    return textWords.map((w, i) => `<span class="word future">${escapeHtml(w)}</span> `).join('');
}
```

## NUI Web Components — internalize before writing any NUI HTML/CSS

This project uses the **NUI Web Components** library (`nui_wc2`). It's a high-performance, browser-native UI toolkit that breaks with many common web development patterns. **Any LLM working on this codebase MUST internalize these rules before writing HTML or CSS.**

### Source of truth (read first, in this order)

1. **Cheatsheet:** [`modules/nui_wc2/LLM-CHEATSHEET.md`](modules/nui_wc2/LLM-CHEATSHEET.md) — every component, exact HTML structure, all built-in actions, addon requirements. No philosophy, just patterns.
2. **Philosophy & Architecture:** [`modules/nui_wc2/documentation/DOCUMENTATION.md`](modules/nui_wc2/documentation/DOCUMENTATION.md) — *why* it works the way it does, only when you need to understand the why.
3. **Component Registry:** [`modules/nui_wc2/documentation/components.json`](modules/nui_wc2/documentation/components.json) — index of all components.
4. **Per-component docs:** [`modules/nui_wc2/documentation/components/{name}.md`](modules/nui_wc2/documentation/components/) — when in doubt about a specific component.
5. **Theme CSS:** [`modules/nui_wc2/NUI/css/nui-theme.css`](modules/nui_wc2/NUI/css/nui-theme.css) — only if you need to read a CSS variable's purpose.
6. **Core JS:** [`modules/nui_wc2/NUI/nui.js`](modules/nui_wc2/NUI/nui.js) — only when debugging a NUI bug at the framework level.
7. **Live reference:** `http://127.0.0.1:5500/modules/nui_wc2/Playground/index.html` — keep this open as a browser tab while working. Visual confirmation is faster than guessing.

### Top rules (in order of importance)

1. **Every `nui-*` element wraps a native HTML element.** Dev auto-creates the inner element; production MUST include it explicitly.
   - `<nui-button><button type="button">Text</button></nui-button>` — NOT `<nui-button>Text</nui-button>`
   - `<nui-input><input type="text"></nui-input>` — NOT `<nui-input type="text">`
   - `<nui-sidebar><nav>...</nav></nui-sidebar>` — NOT `<nui-sidebar>...</nui-sidebar>`
2. **Use `data-action` for declarative wiring**, not scattered event listeners.
   - `data-action="dialog-open@#my-dialog"`
   - `data-action="tabs-select:tabId@#my-tabs"`
3. **`<nui-app>` requires EXACT children in order:**
   `nui-skip-links` → `nui-app-header` → `nui-sidebar` → `nui-content` → (optional `nui-app-footer`). Each layout wrapper MUST contain its native element (`<header>`, `<nav>`, `<main>`, `<footer>`).
4. **NEVER style NUI components.** No inline `style=""`, no `<style>` blocks targeting `nui-*` elements, no custom CSS classes on NUI components. Visual variation comes from attributes (`variant`, `type`, `size`, `fill`). For spacing, use your own wrapper `<div>` with `gap: var(--nui-space)`.
5. **Theme CSS variables only** when you must write CSS: `--color-base`, `--color-shade1-9`, `--text-color`, `--text-color-dim`, `--color-highlight`, `--border-shade1-4`, `--nui-space`, `--nui-space-half`, `--nui-space-double`, `--border-radius1-3`. Never invent new ones.
6. **Addons require BOTH JS import AND CSS link.** Forgetting either = broken component with zero errors. Core components work without imports.
7. **`await nui.ready()`** before calling any programmatic APIs. The library initializes asynchronously.
8. **In page scripts, use `element.querySelector()` not `document.querySelector()`.** The element is the page wrapper passed to `init()`.
9. **Do not listen for `nui-click`** — it's internal. Use `data-action` or listen for the native event on the inner element.

### Decision tree (when tempted to add CSS)

| Want to... | Do this |
|---|---|
| Change a button color | `variant="primary\|outline\|ghost\|danger\|warning"` |
| Make a button look like an icon | `variant="icon"` with `<nui-icon>` inside |
| Change spacing between elements | Your own wrapper `<div>` with `gap: var(--nui-space)` |
| Add a border to something | Check if `<nui-card>` already does it |
| Change font size | `<h1>`–`<h6>` or the `lead` class — NUI handles typography |
| Make something full-width | `nui-page breakout` or `nui-layout` |
| Add elevation/shadow | NUI components handle it — check component docs |
| Tabular data | `<nui-table><table>...</table></nui-table>` |

**Rule of thumb:** If you're reaching for CSS on a NUI component, you're almost certainly using the wrong pattern. Stop and check the component docs.

### Theme variables quick reference

- **Spacing:** `--nui-space` (1rem), `--nui-space-half`, `--nui-space-quarter`, `--nui-space-double`, `--nui-space-eighth`
- **Surfaces (light→dark):** `--color-base` (page bg), `--color-shade1` (slight elevation), `--color-shade2` (card/input bg), `--color-shade3` (hover), `--color-shade4` (active/pressed), `--color-shade5-9` (borders, dim text, disabled)
- **Text:** `--text-color` (primary), `--text-color-dim` (secondary), `--color-text`/`--color-text-dim` (aliases), `--color-text-muted`, `--color-text-subtle`
- **Borders:** `--border-thickness`, `--border-radius1-3`, `--border-shade1-4`
- **Accent:** `--color-highlight` (primary accent for buttons, links, focus), `--color-highlight-dim`, `--color-accent` (alias)
- **Typography:** `--font-size-base` (1rem), `--font-size-xsmall/small/medium/large/xlarge`, `--system-ui`
- **Layout:** `--sidebar-width` (21rem), `--app-header-height` (4rem), `--nui-form-row-height` (2.5rem), `--icon-size` (1.2rem)

### Gotchas specific to this codebase

- **`nui-dialog[mode="page"]`** sets `overflow: auto` on both the dialog and its `<main>` child, which clips absolutely-positioned popovers inside. To allow popovers (e.g. `nui-select` dropdowns) to escape, override to `overflow: visible` on dialog, main, and tabpanel. The source pane can keep its own internal scroll for long content.
- **`nui-list` height caching bug:** the [`checkHeight()`](modules/nui_wc2/NUI/lib/modules/nui-list.js) function runs every 300ms. When the host page is `display: none`, `container.firstChild.offsetHeight` reads as 0, and the function sets `itemHeight = 0`, collapsing the container. Submodule patch: add `if (list.stop) return;` at the top of `checkHeight()`. The `IntersectionObserver` already sets `list.stop = true` when the list is hidden.
- **NUI page cache gotcha:** the router caches page elements at the JavaScript level. `init(element, params, nui)` only runs **once per page per JS context**. Subsequent navigations call `element.show?.(params)`. If the first `init()` throws, the cached element stays broken across reloads. Recovery: navigate to a different page first, or close and re-open the tab. The dev-tools "Disable cache" checkbox prevents this entirely during dev.
- **Slide list `nui-list` does NOT work** for our 58-slide deck — `checkHeight()` runs every 300ms on a hidden page, and `display: none` sets `itemHeight = 0` so the list collapses. The Slides card uses **plain flexbox rows** instead. (See the submodule patch above for a future fix if we want to switch back to `nui-list`.)
- **No `nui-progress` for render progress.** The render endpoint's progress is now surfaced via a banner in the editor; the render page itself just shows the dot status. (The `nui-progress` element is still in render.html but only briefly visible during rendering.)

### What about styling the `nui-list` if we DO use it?

The same rules apply: visual variation via attributes (`variant="primary"`, `size="lg"`, etc.), not CSS. If you need a *nui-list* that doesn't have the cache bug, you'd need the submodule patch applied — which we have, but the Slides card uses flexbox for now to avoid the runtime complexity.
